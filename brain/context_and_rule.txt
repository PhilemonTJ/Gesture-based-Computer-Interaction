## Additional context: ROI tracking + OneEuroFilter integration

The existing project already:

- Captures webcam frames.
- Runs a hand-tracking model (e.g., MediaPipe Hands) that outputs 21 landmarks per hand. MediaPipe’s pipeline already uses a palm detector + hand-landmark model and can run in real time on modest hardware. [research](https://research.google/pubs/mediapipe-hands-on-device-real-time-hand-tracking/)
- Has some gesture logic / virtual-mouse style control, but **does not yet implement**:
  - Region-of-Interest (ROI) tracking around the hand.
  - OneEuroFilter-based temporal smoothing.

Your task is to **augment** the current implementation with these two concepts, not to redesign the entire project.

***

## 1. Region of Interest (ROI) tracking – what to add

**Goal:** Stop running hand detection on the full-resolution frame every time. Instead, track the hand in a small moving window for speed and robustness.

**Required behaviour:**

1. Maintain a small state object for ROI:
   - `roi_active: bool`
   - `roi_box: (x, y, w, h)` in full-frame pixel coordinates (top-left + size)
   - `lost_counter: int` (how many consecutive frames we failed to detect a hand)

2. Per frame, choose the input to the hand model:
   - If `roi_active == False` or `lost_counter` above a threshold:
     - Use the **full frame** as input.
     - After a successful detection, create a new `roi_box` around the detected hand and set `roi_active = True`, `lost_counter = 0`.
   - Else (tracking active):
     - Crop the frame to `roi_box` plus a configurable margin.
     - Run the hand model only on this crop.

3. Coordinate transforms:
   - Hand model landmarks come back in **crop-space** (relative to the cropped image).
   - Convert them back to **full-frame pixel space** by adding the crop’s top-left `(offset_x, offset_y)` and scaling by the crop width/height.

4. ROI update:
   - Compute a new ROI box from the **raw full-frame landmarks**:
     - For example, take min/max of all x,y landmark pixels, then expand by some padding.
   - Clamp ROI to frame bounds.
   - If hand not detected in ROI for N frames:
     - Set `roi_active = False` so the next iteration does a full-frame search.

**Integration expectations:**

- Wrap this as a clean module or class (e.g., `RoiTracker`) with methods like:
  - `get_frame_region(full_frame) -> (roi_frame, offset_x, offset_y)`
  - `update_from_landmarks(landmarks_full_frame)`
  - `mark_lost()` when no hand is found.
- Replace any current “always full-frame” hand detection loop with this ROI-aware logic.

## 2. OneEuroFilter – what to add and where

The project currently uses **raw landmark coordinates** frame-to-frame. That causes jitter for human motion, which the OneEuroFilter is designed to address: a simple, adaptive low-pass filter that reduces jitter at low speeds while keeping responsiveness at high speeds. [mohamedalirashad.github](https://mohamedalirashad.github.io/FreeFaceMoCap/2021-12-25-filters-for-stability/)

**Requirements:**

1. Add a OneEuroFilter implementation (or use an existing small Python implementation):
   - It should expose a callable interface:
     - `filtered_value = filter_instance(t, raw_value)`
   - It must keep internal state (previous value, previous derivative, previous timestamp).
   - Support two tunable parameters:
     - `min_cutoff` – baseline smoothing (jitter reduction).
     - `beta` – how much smoothing is reduced at higher speeds (to reduce lag).

2. Instantiate filters for each continuous control dimension:
   - At minimum, create:
     - `filter_x` for x-coordinate of the main control point (e.g., index fingertip).
     - `filter_y` for y-coordinate.

3. IMPORTANT: Apply filtering in **full-frame/screen space**, after ROI coordinate conversion:
   - Compute raw full-frame pixel coordinates from hand landmarks (after undoing the crop).
   - Pass those raw screen coordinates and current time to OneEuroFilter.
   - Use the **filtered** output for:
     - Cursor movement.
     - Features that depend on stable position (e.g., velocity, dwell-based gestures), if needed.

4. Do **not** use filtered coordinates to update the ROI:
   - ROI should track the **true** hand position; use **raw** landmark-derived positions to update the bounding box.
   - This avoids the ROI box lagging behind the actual hand.

**Integration expectations:**

- Introduce a small module or class like `filters.py` with `OneEuroFilter`.
- Hook it into the main loop where cursor coordinates or gesture features are computed:
  - Drop-in replacement: instead of using `raw_x, raw_y` directly, pass them through the filter and use `smooth_x, smooth_y` for UI.

## 3. How you should analyse the current project

When given the user’s repo / code, do a focused pass to:

1. **Identify the existing detection loop:**
   - Find where webcam frames are read.
   - Find where the hand tracking model (likely MediaPipe Hands or equivalent) is called.
   - Note whether it always uses the full frame.

2. **Locate how landmarks flow through the system:**
   - Where are the 21 landmarks first available?
   - How are they converted to screen space?
   - Where is cursor movement or gesture recognition done?

3. **Check for any current smoothing:**
   - Is there a moving average, simple low-pass, or no smoothing at all?
   - If something exists, be ready to replace or augment it with OneEuroFilter.

4. **Propose exact insertion points:**
   - Before calling the hand model → insert ROI selection (crop vs full-frame).
   - Right after receiving landmarks → insert:
     - ROI update (using raw full-frame positions).
     - OneEuroFilter calls (for positions used by the UI / gestures).

5. **Keep everything else intact:**
   - Gesture logic, classifier, and other app logic should not be disrupted, just fed smoother, more efficient landmark data.
